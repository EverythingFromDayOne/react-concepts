### Recipe #9 — auth track opener: single-flight token refresh

| # | Recipe | Status | Scenario |
| --- | --- | --- | --- |
| 9 | `recipes/auth/refresh-storm-on-401.md` | 🟢 | Dashboard loads 6 widgets in parallel; access token expired during a 20-min idle tab; all 6 → 401 at once → 6 concurrent `/auth/refresh` calls. With refresh-token **rotation** (security best practice), the first refresh consumes+rotates the token, the other 5 present the now-dead token → rejected → app reads it as "session dead" → logout. ~1-in-3 post-idle loads; "it keeps logging me out." Escaped QA: tests issue requests sequentially (never collide) + rotation often only in staging/prod + needs an expired-token-at-multi-request-load. Fix arc: name it (refresh is per-request, should be per-app; rotation makes concurrent refreshes self-invalidate) → reject the naive `isRefreshing` boolean (stops extra refreshes but drops the requests that should wait/retry) → **single-flight refresh with a shared promise** (`refreshPromise ??= refreshToken().finally(()=>refreshPromise=null)`; first 401 starts it, all concurrent 401s await the SAME one, each retries its own request once) → fail-once (idempotent `onAuthFailure` = `queryClient.clear()` + redirect once, not per-request) + integrate (authFetch under Query/Router; Query `retry:false` on 401 so it doesn't 3× the storm). The auth-track equivalent of search-race (dedup concurrent in-flight work). 11 pitfalls + when-NOT (opaque server-side sessions / single-request pages). httpOnly-cookie + in-memory-access-token security stance; CSRF note. Verify-the-loop: 6 parallel calls → assert 1 refresh + 0 logouts |

| # | Recipe | Status | Scenario |
| --- | --- | --- | --- |
| 11 | `recipes/auth/logout-leaves-stale-cache.md` | 🟢 | Shared-workstation support console; agent A logs out (token cleared, `navigate('/login')`) but the root `QueryClient` lives for the tab's life, so its cache still holds A's `['tickets','queue']`/`['customer',id]` PII; agent B logs in → protected routes mount → cache-hit renders A's data to B (a cross-user data-exposure incident, not a flicker). ## Locked editorial conventions

### Article structure (concept articles) — v2, July 2026 depth revision

1. Frontmatter (`article_id`, `concept_folder`, `wave`, `related`, `react_baseline`, `status`)

### Code conventions

- Function components only; classes appear once (error-boundaries article), immediately wrapped by `react-error-boundary`
- TypeScript strict throughout; no `React.FC` — plain function with a typed props interface; discriminated unions for variant props
- Effects contract: every subscription-like effect returns a cleanup; every fetch inside an effect uses `AbortController` and ignores aborts; effects synchronize with external systems — never derive state
- Derive during render, never mirror props into state (`useEffect`-sets-state shown only as the anti-pattern)
- Keys are stable IDs (`key={item.id}`), never array index
- `ref` as a prop (19+) over `forwardRef`; legacy form shown with marker
- Compiler-first memoization: pure components, Compiler assumed ON for app code; `memo`/`useMemo`/`useCallback` taught as mechanism, reserved for library code and non-compiled boundaries — no cargo-cult memoization in examples
- State placement rule: local UI state → `useState`/`useReducer`; server state → TanStack Query (never in a client store); cross-cutting client state → Zustand; static config → context
- Mutations: form actions + `useActionState`/`useOptimistic` where forms are involved; TanStack Query mutations elsewhere
- Testing: Vitest + RTL + user-event + MSW; query by role first, `data-testid` fallback only
- Named exports; co-located test files (`Component.test.tsx`)
